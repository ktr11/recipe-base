# 目的
Next.js (App Router) と AWS Amplify Gen 2 を使用して、レシピ保存・家族共有Webアプリを構築してください。画面のスタイリングには Tailwind CSS と daisyUI を使用します。

# 技術スタック
- Frontend: Next.js (App Router, TypeScript)
- UI Library: Tailwind CSS + daisyUI
- Backend/Infrastructure: AWS Amplify Gen 2 (Auth, Data)
- Auth: AWS Cognito
- Database: Amazon DynamoDB (Amplify Data で管理)

# インフラ管理（IaC）方針
- インフラは AWS Amplify Gen 2 の標準（AWS CDKベース of TypeScript IaC）に従って管理します。
- CloudFormation テンプレートを直接書いたり、AWS コンソールで手動構築したりせず、すべて `amplify/auth/resource.ts` や `amplify/data/resource.ts` などのコード定義からプロビジョニングを行ってください。

# 機能要件・仕様

## 1. 認証・ユーザー管理
- **サインアップ/サインイン:** メールアドレスとパスワードによる認証。
- **ユーザー設定:** ログイン後に「ユーザー名変更」「パスワード変更」ができること。
- **カラーテーマ設定:** ユーザーごとに選択したテーマを保存・適用できること。

## 2. トライアルログイン（未認証ゲスト機能）
- ログインせずに「ゲストとして利用」できるボタンを配置。
- AWS Amplify Gen 2 の Guest アクセス権限（Cognito Unauthenticated Identity）を使用し、未認証状態でもデータを AWS（DynamoDB）に保存する。
- **トライアル制限ロジック（重要）:**
  - 作成できるレシピは最大 3 つまで。
  - 作成できるラベルは最大 3 つまで。
  - 1つのレシピに登録できる材料は最大 10 個まで。
  - 制限を超えようとした場合は、制限に達した旨のアラートと、正規のユーザー登録を促す導線を表示すること。

## 3. 家族間レシピ共有（チーム機能） ※正規ユーザー限定機能
- ログイン済みの正規ユーザーのみが利用できる機能。
- **チームの作成と参加:** 
  - ユーザーは新しい「チーム」を作成し、共有用の「招待コード」を発行できる。
  - 他のユーザーはその「招待コード」を入力することで、同じチームに所属できる。
- **共有範囲:** チームに参加すると、チームメンバーが登録したレシピやカスタムラベルが自動的に同期され、メンバー全員が閲覧・編集・削除できるようになる。

## 4. レシピ・ラベル管理機能
- **レシピの項目:**
  - 料理名
  - レシピサイトURL（任意）
  - 材料・分量
  - ラベル（複数付与可能。例：主菜、野菜、サラダ、炭水化物、スープ）
  - メモ（フリーテキスト）
- **ラベル管理:** ユーザー単位（ゲストはセッション/Identity単位、チーム所属時はチーム単位）でラベルを自由に作成・削除できること。
- **レシピ検索:** 「料理名」「材料名（部分一致）」「選択したラベル」でフィルタリング検索ができること。

## 5. UI/UX・デザイン
- **カラーテーマ:** daisyUI のテーマ切り替え機能（data-theme）を利用し、以下の4テーマを切り替え可能にする。
  1. ライト ブルー (light系ベース)
  2. ライト イエロー (cupcake/retro系ベース)
  3. ダーク ブルーブラック (night系ベース)
  4. ダーク ブラック (luxury/dark系ベース)
- レシピ一覧、作成・編集フォーム、検索バー、チーム管理画面を分かりやすく配置した、クリーンでレスポンシブなUI。

# 開発ステップの指示

ユーザーからGOサインが出るまでは、実際のコード生成やファイル作成を行わず、以下のステップ順にプロセスを進めてください。

## フェーズ1：設計とドキュメント作成（※現在ここを実行してください）
まずは実装を始める前に、設計書として以下のドキュメントを作成し、Markdown形式で出力してください。
1. **データモデル設計 (Amplify Data Schema)**
   - レシピ（Recipe）、ラベル（Label）、およびチーム（Team/Group）のスキーマ構成、リレーションシップの定義案。
   - owner（個人）、guest（未認証）、そして dynamic group（チームメンバー）の認可ルール（Authorization Rules）を Amplify Gen 2 でどう表現するかの設計案。
2. **招待コードによるチーム参加の処理フロー**
   - 招待コードの生成、照合、およびユーザーへのチームID紐付け（Cognitoのカスタム属性やDynamoDBでのマッピング）の具体的な仕組み。
3. **画面遷移・ルーティング設計**
   - 必要なページ（例: `/`, `/recipes`, `/settings`, `/team` など）と、それぞれのページが「ゲスト利用可能」か「要ログイン」かの定義。
4. **トライアル制限のバリデーション方針**
   - ゲストユーザーがレシピやラベルを作成する際、上限（3つ/3つ/10個）をどのコンポーネントまたはフックで判定・ブロックするかの詳細な処理フロー。

**【注意】このドキュメントを出力した時点で一度回答を止め、ユーザーの確認とフィードバックを求めてください。ユーザーから「実装を開始してください」と指示があるまで、フェーズ2のコード作成には進まないでください。**

## フェーズ2：インフラ・コード実装（ユーザーのGOサイン後に開始）
1. `amplify/auth/resource.ts` でゲストアクセスおよびチーム属性等を考慮した Auth 設定を構築してください。
2. `amplify/data/resource.ts` で、フェーズ1で合意したチーム共有対応のスキーマを定義してください。
3. `tailwind.config.js` と `app/layout.tsx` に daisyUI のテーマ設定を組み込み、テーマ切り替えコンポーネントを作成してください。
4. トライアル制限を判定するカスタムフックまたはユーティリティを実装し、保存処理の直前でバリデーションをかけるロジックを組み込んでください。
5. チーム作成・招待コード参加機能、および各画面のフロントエンド実装を進めてください。
